    1 Writing a Device Driver in RIOT {#driver-guide}
    2 ===============================
    3 
    4 This document describes the requirement, design objectives, and some
    5 non-function details when writing device drivers in/for RIOT. The term device
    6 driver in this context includes all 'CPU-external' devices connected to the CPU
    7 typically via peripherals like SPI, I2C, UART, GPIO, and similar. CPU
    8 peripherals itself are in RIOT not considered to be device drivers, but
    9 peripheral or low-level drivers. Typical devices are network devices like
    10 radios, Ethernet adapters, sensors, and actuators.
    11 
    12 [TOC]
    13 
    14 # General Design Objectives {#driver-guide-design-objectives}
    15 
    16 Device drivers should be as easy to use as possible. This implies an
    17 'initialize->ready' paradigm, meaning, that device drivers should be ready to use
    18 and in operation right after they have been initialized. On top, devices should
    19 work with physical values wherever possible. So e.g. sensors should return
    20 already converted values in some physical unit instead of RAW data, so that
    21 users can work directly with data from different devices directly without having
    22 to care about device specific conversion.
    23 
    24 However, please avoid the use of `float` or `double`. Instead, multiply to the
    25 next SI (or appropriate) unit. E.g. if an ADC would return values like `1.23 V`,
    26 chose to return `1230 mV` instead.
    27 
    28 Additionally towards ease of use, all device drivers in RIOT should provide a
    29 similar 'look and feel'. They should behave similar concerning things like their
    30 state after initialization, like their used data representation and so on.
    31 
    32 Secondly, all device drivers should be optimized for minimal RAM/ROM usage, as
    33 RIOT targets (very) constrained devices. This implies, that instead of exposing
    34 all thinkable functionality, the drivers should focus on exporting and
    35 implementing a device's core functionality, thus covering ~95% of the use cases.
    36 Furthermore great care should be put into ...(?)
    37 
    38 Third, it should always be possible, to handle more than a single device of one
    39 kind. Drivers and their interfaces are thus designed to keep their state
    40 information in a parameterized location instead of driver defined global
    41 variables.
    42 
    43 Fourth, RIOT defines high-level interfaces for certain groups of devices (i.e.
    44 netdev for network devices, SAUL for sensors and actuators), which enable users
    45 to work with a wide variety of devices without having to know anything about the
    46 actual device that is mapped.
    47 
    48 Fifth, during initialization we make sure that we can communicate with a device.
    49 Other functions should check the dev pointer is not void, and should also handle
    50 error return values from the lower layer peripheral driver implementations,
    51 where there are some.
    52 
    53 Sixth, device drivers SHOULD be implemented independent of any CPU/board code.
    54 To achieve this, the driver implementations should strictly be based on
    55 platform independent interfaces as the peripheral drivers, xtimer, etc.

    57 # General {#driver-guide-general}
    58 
    59 ## Documentation {#driver-guide-doc}
    60 
    61 Document what your driver does! Most devices come with a very large number of
    62 features, while the corresponding device driver only supports a sub-set of them.
    63 This should be clearly stated in the device driver's documentation, so that
    64 anyone wanting to use the driver can find out the supported features without
    65 having to scan through the code.
    66 
    67 ## Device descriptor and parameter configuration {#driver-guide-types}
    68 
    69 Each device MUST supply a data structure, holding the devices state and
    70 configuration, using the naming scheme of `DEVNAME_t` (e.g. `dht_t`, or
    71 `at86rf2xx_t`). In the context of RIOT, we call this data structure the device
    72 descriptor.
    73 
    74 This device descriptor MUST contain all the state data of a device. By this, we
    75 are not limited on the number of instances of the driver we can run
    76 concurrently. The descriptor is hereby used for identifying the device we want
    77 to interact with, and SHOULD always be the first parameter for all device driver
    78 related functions.
    79 
    80 Typical things found in these descriptors are e.g. used peripherals (e.g. SPI or
    81 I2C bus used, interfacing GPIO pins), data buffers (e.g. RX/TX buffers where
    82 needed), or state machine information.
    83 
    84 On top of the device descriptor, each device driver MUST also define a data
    85 structure holding the needed configuration data. The naming scheme for this type
    86 is `DEVNAME_params_t`. In contrary to the device descriptor, this data structure
    87 should only contain static information, that is needed for the device
    88 initialization as it is preferably allocated in ROM.
    89 
    90 A simple I2C temperature sensors's device descriptor could look like this:
    91 
    92 @code{.c}
    93 typedef struct {
    94  tmpabc_params_t p; /**< device configuration parameter like I2C bus and bus addr */
    95  int scale; /**< some custom scaling factor for converting the results */
    96 } tmpabc_t;
    97 
    98 /* with params being */
    99 typedef struct {
    100  i2c_t bus; /**< I2C bus the device is connected to */
    101  uint8_t addr; /**< the device's address on the bus */
    102 } tmpabc_params_t;
    103 @endcode
    104 
    105 **NOTE:** In many cases it makes sense, to copy the `xxx_params` data into the
    106 device descriptor during initialization. In some cases, it is however better to
    107 just link the `params` data via pointer and only copy selected data. This way,
    108 configuration data that is only used once can be read directly from ROM, while
    109 often used fields (e.g. used peripherals) are stored directly in the device
    110 descriptor and one saves hereby one de-referencing step when accessing them.
    111 
    112 ## Default device configuration {#driver-guide-config}
    113 
    114 Each device driver in RIOT MUST supply a default configuration file, named
    115 `DEVNAME_params.h`. This file should be located in the `RIOT/drivers/...`. The
    116 idea is, that this file can be overridden by an application or a board, by
    117 simply putting a file with the same name in the application's or the board's
    118 include folders, while RIOT's build system takes care of preferring those files
    119 instead of the default params file.
    120 
    121 A default parameter header file for the example temperature sensor above would
    122 look like this (`tmpabc_params.h`):
    123 
    124 @code{.c}
    125 /* ... */
    126 
    127 #include "board.h" /* THIS INCLUDE IS MANDATORY */
    128 #include "tmpabc.h"
    129 
    130 /* ... */
    131 
    132 /**
    133  * @brief Default configuration parameters for TMPABC sensors
    134  * @{
    135  */
    136 #ifndef TMPABC_PARAM_I2C
    137 #define TMPABC_PARAM_I2C (I2C_DEV(0))
    138 #endif
    139 #ifndef TMPABC_PARAM_ADDR
    140 #define TMPABC_PARAM_ADDR (0xab)
    141 #endif
    142 
    143 #ifndef TMPABC_PARAMS
    144 #define TMPABC_PARAMS { .i2c = TMPABC_PARAM_I2C \
    145  .addr = TMPABC_PARAM_ADDR }
    146 #endif
    147 /** @} */
    148 
    149 /**
    150  * @brief Allocation of TMPABC configuration
    151  */
    152 static const tmpabc_params_t tmpabc_params[] = {
    153  TMPABC_PARAMS
    154 }
    155 /* ... */
    156 @endcode
    157 
    158 Now to influence the default configuration parameters, we have these options:
    159 
    160 First, we can override one or more of the parameter from the makesystem, e.g.
    161 
    162 @code{.sh}
    163 CFLAGS="-DTMPABC_PARAM_ADDR=0x23" make all
    164 @endcode
    165 
    166 Second, we can override selected parameters from the board configuration
    167 (`board.h`):
    168 
    169 @code.{c}
    170 /* ... */
    171 /**
    172  * @brief TMPABC sensor configuration
    173  * @{
    174  */
    175 #define TMPABC_PARAM_I2C (I2C_DEV(1))
    176 #define TMPABC_PARAM_ADDR (0x17)
    177 /** @} */
    178 /* ... */
    179 @endcode
    180 
    181 Third, we can define more than a single device in the board configuration
    182 (`board.h`):
    183 
    184 @code{.c}
    185 /* ... */
    186 /**
    187  * @brief Configure the on-board temperature sensors
    188  * @{
    189  */
    190 #define TMPABC_PARAMS { \
    191  .i2c = I2C_DEV(1), \
    192  .addr = 0x31 \
    193  }, \
    194  { \
    195  .i2c = I2C_DEV(1), \
    196  .addr = 0x21 \
    197  }
    198 /** @} */
    199 /* ... */
    200 @endcode
    201 
    202 And finally, we can simply override the `tmpabc_params.h` file as described
    203 above.

    205 ## Initialization {#driver-guide-initialization}
    206 
    207 In general, the initialization functions should to the following:
    208 
    209 - initialize the device descriptor
    210 - initialize non-shared peripherals they use, e.g. GPIO pins
    211 - test for device connectivity, e.g. does a SPI/I2C slave react
    212 - reset the device to a well defined state, e.g. use external reset lines or do
    213  a software rest
    214 - do the actual device initialization
    215 
    216 For testing a device's connectivity, it is recommended to read certain
    217 configuration data with a defined value from the device. Some devices offer
    218 `WHO_AM_I` or `DEVICE_ID` registers for this purpose. Writing and reading back
    219 some data to the device is another valid option for testing it's responsiveness.
    220 
    221 For more detailed information on how the signature of the init functions should
    222 look like, please refer below to the specific requirements for network devices
    223 and sensors.
    224 
    225 ## Return values {#driver-guide-return-values}
    226 
    227 As stated above, we check communication of a device during initialization, and
    228 handle error return values from the lower layers, where they exist. To prevent
    229 subsequent misuse by passing NULL pointers and similar to the subsequent
    230 functions, the recommended way is to check parameter using `assert`, e.g.:
    231 
    232 @code{.c}
    233 int16_t tmpabc_read(const tmpabc_t *dev)
    234 {
    235  assert(dev);
    236  /* ... */
    237  return value;
    238 }
    239 @endcode
    240 
    241 Whenever status/error return values are implemented by you in your driver, they
    242 should be named, meaning that the driver MUST define an enum assigning names to
    243 the actual used value, e.g.
    244 
    245 @code{.c}
    246 enum {
    247  TMPABC_OK = 0, /**< all went as expected */
    248  TMPABC_NOI2C = -1, /**< error using the I2C bus */
    249  TMPABC_NODEV = -2 /**< no device with the configured address found on the bus */
    250 };
    251 @endcode

    253 ## General Device Driver Checklist {#driver-guide-general-checklist}
    254 
    255 - *MUST*: the supported feature set and any custom behavior is clearly
    256  documented
    257 - *MUST*: device descriptor is defined: `devab_t`
    258 - *MUST*: device parameter `struct` is defined: `devab_params_t`
    259 - *MUST*: a default parameter configuration file is present, e.g.
    260  `RIOT/drivers/devab/include/devab_params.h`
    261 - *MUST*: all error and status return codes are named, e.g.
    262  `DEVAB_OK, DEVAB_NOSPI, DEVAB_OVERFLOW, ...`
    263 - *MUST*: use `const devab_t *dev` when the device descriptor can be access
    264  read-only

    266 ## Helper tools
    267 
    268 To help you start writing a device driver, the RIOT build system provides the
    269 `generate-driver` make target. It is a wrapper around the
    270 [riotgen](https://pypi.org/project/riotgen/) command line tool that is helpful
    271 when starting to implement a driver: all minimum files are generated with
    272 copyright headers, doxygen groups, etc, so you can concentrate on the driver
    273 implementation.
    274 
    275 **Usage:**
    276 
    277 From the RIOT base directory, run:
    278 ```
    279 make generate-driver
    280 ```
    281 Then answer a few questions about the driver:
    282 - Driver name: enter a name for your driver. It will be used as both the name
    283  of the driver directory where the source files are created and the build
    284  system module.
    285 - Driver doxygen group name: Enter the name of driver, as displayed in the
    286  Doxygen documentation.
    287 - Brief doxygen description: Describe in one line what is this driver about.
    288 - Parent driver Doxygen group: Enter the Doxygen group the driver belongs to.
    289  It can be `actuators`, `display`, `misc`, `netdev`, `sensors`, `storage`.
    290 
    291 Other global information (author name, email, organization) should be retrieved
    292 automatically from your git configuration.

    294 # Sensors {#driver-guide-sensors}
    295 
    296 ## SAUL {#driver-guide-saul}
    297 
    298 All sensor drivers SHOULD implement the SAUL interface. It is however
    299 recommended, that the drivers are written in a way, that the drivers do not
    300 solely export the SAUL interface, but map the SAUL interface upon a driver
    301 specific one.
    302 
    303 For example, a temperature driver provides the following function (`tmpabc.c`):
    304 
    305 @code{.c}
    306 int16_t tmpabc_read(tmpabc_t *dev);
    307 @endcode
    308 
    309 which then can easily be mapped to SAUL (`tmpabc_saul.c`):
    310 
    311 @code{.c}
    312 int saul_read(saul_t *dev, phydat_t *data)
    313 {
    314  memset(data, 0, sizeof(phydat_t));
    315  data->x = tmpabc_read((tmpabc_t *)dev);
    316  data->unit = UNIT_TEMP_C;
    317  data->scale = -2;
    318  return 1;
    319 }
    320 @endcode
    321 
    322 This ensures the versatility of the device driver, having in mind that one might
    323 want to use the driver without SAUL or maybe in a context without RIOT.
    324 
    325 ## Initialization {#driver-guide-sensor-initialization}
    326 
    327 Sensor device drivers are expected to implement a single initialization
    328 function, `DEVNAME_init`, taking the device descriptor and the device's
    329 parameter struct as argument:
    330 
    331 @code{.c}
    332 int tmpabc_init(tmpabc_t *dev, const tmpabc_params_t *params);
    333 @endcode
    334 
    335 After this function is called, the device MUST be running and usable.
    336 
    337 ## Value handling {#driver-guide-sensor-value-handling}
    338 
    339 ### Value semantics {#driver-guide-sensor-value-semantics}
    340 
    341 All sensors in RIOT MUST return their reading results as real physical values.
    342 When working with sensor data, these are the values of interest, and the
    343 overhead of the conversion is normally neglectable.
    344 
    345 ### Typing {#driver-guide-sensor-types}
    346 
    347 All values SHOULD be returned using integer types, with `int16_t` being the
    348 preferred type where applicable.
    349 
    350 In many situations, the physical values cannot be mapped directly to integer
    351 values. For example, we do not want to map temperature values to integers
    352 directly while using their fraction. The recommended way to solve this is by
    353 scaling the result value using decimal fixed point arithmetic, in other words
    354 just return centi-degree instead of degree (e.g. 2372cC instead of 23.72C).
    355 
    356 ## Additional Sensor Driver Checklist {#driver-guide-sensor-checklist}
    357 
    358 - *MUST*: mandatory device parameters are configurable through this file, e.g.
    359  sampling rate, resolution, sensitivity
    360 - *MUST*: an init function in the style of
    361  `int devab_init(devab_t *dev, const devab_params_t *params);` exists
    362 - *MUST*: after initialization, the device must be operational
    363 - *MUST*: all error and return values are named, e.g.
    364  `DEVAB_OK, DEVAB_NODEV, ...`
    365 - *MUST*: all 'read' functions return values in their physical representation,
    366  e.g. `centi-degree, Pa, lux, mili-G`
    367 - *MUST*: all 'read' functions return integer values, preferably `int16_t`
    368 - *SHOULD*: if multiple dimensions are read, they SHOULD be combined into a
    369  data structure
    370 - *SHOULD*: the driver implements the SAUL interface
    371 - *SHOULD*: the driver exports functions for putting it to sleep and waking up
    372  the device

    374 # Network devices {#driver-guide-netdev}
    375 
    376 ## Initialization {#driver-guide-netdev-init}
    377 
    378 The initialization process MUST be split into 2 steps: first initialize the
    379 device descriptor and if applicable the used peripherals, and secondly do the
    380 actual device initialization. The reason for this is, that before a device is
    381 actually activated and can start to process data, the network stack for the
    382 device needs to be initialized. By supplying a second init function, that does
    383 the actual initialization, we can hand the control over when this is done to the
    384 actual network stacks.
    385 
    386 The initialization functions SHOULD follow this naming scheme:
    387 
    388 @code{.c}
    389 void netabc_setup(netabc_t *dev, const netabc_params_t *params);
    390 int netabs_init(netabc_t *dev);
    391 @endcode
    392 
    393 ## netdev {#driver-guide-netdev-interface}
    394 
    395 Device driver for network device SHOULD implement the `netdev` interface. It is
    396 up to the implementer, if the device driver also offers a device specific
    397 interface which is then mapped to the `netdev` interface, or if the device
    398 driver can be purely interfaced using `netdev`. While the second option is
    399 recommended for efficiency reasons, this is not mandatory.
    400 
    401 ## Additional Network Device Driver Checklist {#driver-guide-netdev-checklist}
    402 
    403 - *MUST*: a setup function in the style of
    404  `int devab_setup(devab_t *dev, const devab_params_t *params);` exists
    405 - *MUST*: an init function in the style of `int devnet_init(devnet_t *dev)`
    406  exists
    407 - *SHOULD*: the driver implements 'netdev' [if applicable]
